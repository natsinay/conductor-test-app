Conductor platform

Every change takes the same well-lit path.

Conductor moves a change through a fixed pipeline. Most stages are gates that can stop the run; one is advisory and only offers guidance. The same pipeline.js module that renders the map below is unit tested with node --test — no build step, no dependencies.

The pipeline at a glance

Stages run top to bottom. A failing gate halts everything after it; advisory review never blocks.

  1. Spec Gate
  2. Build
  3. Advisory Review
  4. Execution Gate
  5. Merge Gate
  6. Promote

What each stage does

1 · Spec Gate Gate

Before any work begins, the spec gate confirms the change has an approved specification. No approved spec, no pipeline — this keeps work anchored to something the team actually agreed to build.

2 · Build Gate

The change is compiled and the automated test suite runs. A red build is a hard stop: nothing downstream sees a change that does not compile or pass its tests.

3 · Advisory Review Advisory

The only non-blocking stage. Automated and human reviewers surface suggestions, style notes, and risks. These annotate the run so people can weigh them, but they never stop it — advisories inform, they do not gate.

4 · Execution Gate Gate

Approval to run the change against real, live resources. This is the boundary between "safe to try" and "touching production data," so it requires an explicit go-ahead.

5 · Merge Gate Gate

The final human sign-off before the change lands on the main line. Once merged, the change becomes part of the shared history everyone builds on.

6 · Promote Gate

The merged change is rolled out to the production environment. Promotion is the last gate — it is where a reviewed, merged change becomes something users actually run.
